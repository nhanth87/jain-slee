# micro-jainslee — SBB Session Management & Routing Investigation

> **Status:** R&D only — never for production.
> **Scope:** How a single in-flight USSD/SMS session is anchored to an SBB
> entity, how incoming messages from an RA reach that entity, how the
> entity is re-created or torn down when the virtual-thread slot is
> recycled, and which edge cases the current code does NOT cover.
> **Reference code:** Perfect Core S1–S5 (2026-06-28) on
> `jainslee-core`, `jainslee-cluster`, `jainslee-ra-spi`, and the
> `example-quarkus` USSD demo.
> **Methodology:** line-by-line walk of every method that touches the
> routing/session lifecycle (no abstract advice — every claim cites a
> concrete line range).

---

## 0. Glossary

| Term | Meaning in this report |
|---|---|
| **Session** | One logical USSD/SMS dialog tied to a single convergence key (e.g. `msisdn:dialogId`). |
| **SBB entity** | The runtime instance that owns session state. In micro-jainslee this is a single `Sbb` POJO + a `SimpleSbbLocalObject` + an `EntitySlot` (per-SBB virtual thread). |
| **Entity slot** | `EntitySlot` (`jainslee-core/.../EntitySlot.java`) — long-lived virtual-thread shell that is bound/unbound to fresh SBB POJOs across session lifecycles. Pooled in `EntitySlotPool`. |
| **ACI** | Activity Context Interface — `InMemoryActivityContext` (or the cluster variant). The handle the RA / RA endpoint passes to the SLEE per session. |
| **ACNF** | Activity Context Naming Facility — string→ACI map. In-memory or Infinispan-backed. |
| **Convergence key** | The IES return value (`InitialEventSelectResult.convergenceName`). Used to route subsequent events of the same session to the same SBB entity. |
| **IES** | Initial Event Selector — the per-SBB `@InitialEventSelect` method. Decides whether an incoming event opens a new session or continues an existing one. |
| **Snapshot** | `SbbEntitySnapshot` — a POJO copy of one entity's CMP state held in the Infinispan `sbb-entity-state` cache (cluster mode only). |

## 1. Current state — what works, what breaks

### 1.1 What works (verified at the line level)

1. **Per-SBB virtual-thread pinning.** Every active SBB entity owns one
   parked virtual thread inside `EntitySlot` (`EntitySlot.java:36-46`,
   `EntitySlot.EventLoop.run()` at `EntitySlot.java:117-170`). The slot
   is reused after `unbind()` — see `EntitySlot.unbind()` at
   `EntitySlot.java:94-100` and `EntitySlotPool.release()` at
   `EntitySlotPool.java:57-66`. The JAIN-SLEE spec invariant
   "single-threaded per SBB" holds for as long as the slot exists.

2. **Initial-event convergence via IES.** `InitialEventSelectorDispatcher`
   (`jainslee-core/.../ies/InitialEventSelectorDispatcher.java:97-163`)
   resolves the target entity id from a convergence key:
   * First call (`isInitialEvent=true`) → `pool.allocateNew()`
     → `entities.put(convergence, newId)` (`InitialEventSelectorDispatcher.java:147-150`).
   * Subsequent calls (`isInitialEvent=false`, matching key) → lookup
     returns the existing `newId` (`InitialEventSelectorDispatcher.java:124-130`).
   * When the entity is removed, the convergence entry is auto-cleared
     via `pool.onEntityRemoved(...)` registered at
     `InitialEventSelectorDispatcher.java:156-159`.
   * Unit-tested at
     `jainslee-core/src/test/java/com/microjainslee/core/ies/InitialEventSelectorDispatcherTest.java`
     (10 tests, all pass).

3. **RA→SLEE event routing via Disruptor.**
   `SleeEndpointImpl.fireEvent()` (`SleeEndpointImpl.java:185-229`)
   validates the state machine, then forwards through
   `EventRouterPort → MicroSleeContainer.routeEvent → EventRouter.routeEvent`.
   The LMAX Disruptor ring at `EventRouter.java:45-47` serializes all
   dispatch into one consumer thread.

4. **Entity attach/detach on ACI.**
   `MicroSleeContainer.attach()` at `MicroSleeContainer.java:1003-1021`
   binds the SBB local object onto the ACI and registers a timer
   bridge binding; `detachFromAllActivityContexts()` at
   `MicroSleeContainer.java:1023-1030` does the reverse during
   removal.

5. **Cluster snapshot / reconstruct (cross-node failover, partial).**
   `DistributedSbbEntityPool.acquire(sbbId, factory)` at
   `DistributedSbbEntityPool.java:147-167` looks up the
   `SbbEntitySnapshot` in the `sbb-entity-state` Infinispan cache
   (REPL_ASYNC) and reconstructs the SBB by reflectively writing the
   snapshot back through `@CmpField` setters (`applySnapshot` at
   `DistributedSbbEntityPool.java:283-321`).

6. **Cascade removal of child SBBs.**
   `CascadeRemover.cascadeRemove()` at
   `CascadeRemover.java:115-159` walks the parent→child index
   depth-first post-order, calls `sbbRemove()` on every child, and
   releases the entity back to the pool via `EntityLookup`.

7. **Per-event transaction + error policy.**
   `EventRouter.dispatchWithTransaction()` at
   `EventRouter.java:368-444` opens a logical transaction per event,
   applies the `EventMask` (`acceptsEvent` at `EventRouter.java:459-466`),
   dispatches in priority order, and routes exceptions through
   `handleSbbException` at `EventRouter.java:601-613`.

8. **Session timer armed at HTTP begin.**
   `HttpServerSbb.onHttpBegin()` at
   `example-quarkus/.../sbbs/HttpServerSbb.java:61-75` calls
   `wiring.container().getTimerPort().setTimer(timeout, httpLocal)` and
   stashes the returned timer id in `UssdSbbWiring.sessionTimers`
   (`UssdSbbWiring.java:93-99`). On `TimerFiredEvent`,
   `HttpServerSbb.onTimer()` at `HttpServerSbb.java:82-93` derives the
   session id from the entity id (`"<sid>/http"`) and calls
   `wiring.failSession(...)`.

### 1.2 What is fragile / does NOT work

| # | Failure mode | Where it bites | Severity |
|---|---|---|---|
| 1 | **SBB death between two events silently strands the session.** | `EventRouter.deliverEvent()` at `EventRouter.java:479-548` — if `pool.findEntity(sbbId)` returns `null`, the router falls through to inline `handler.onEvent(...)` on the Disruptor worker thread (`EventRouter.java:483-489`) instead of re-creating the slot. CMP state is gone (the slot has been recycled and rebound for another session). | 🔴 critical |
| 2 | **No RA→SBB back-channel on entity removal.** | `VirtualThreadSbbEntityPool.release*` (`VirtualThreadSbbEntityPool.java:200-217`) and the `RemovalListener` lambda in `MicroSleeContainer.buildLocalObject()` (`MicroSleeContainer.java:1322-1359`) remove the entity from the `sbbs` map and `entityTypesById`, but they never notify RAs that the handle they cached is now invalid. Subsequent HTTP/SIP/SS7 traffic will route to a dead ACI or fail IES lookup. | 🔴 critical |
| 3 | **Out-of-order messages are silently dropped, not queued.** | `InitialEventSelectorDispatcher.resolveTarget()` returns `null` when the convergence is missing AND `isInitialEvent=false` (`InitialEventSelectorDispatcher.java:139-144`, spec §7.5.5). The event is then handed to `EventRouter.routeEvent` with no entity attached — `dispatchWithTransaction` walks `activityContext.getAttachedSbbs()` (`EventRouter.java:383`) and finds an empty list, so the event is dropped with only a debug log. | 🟠 major |
| 4 | **Duplicate / retried messages are processed twice.** | There is no per-session sequence-number check anywhere on the routing path. `EventRouter` does not consult an event id; `SleeEndpointImpl.fireEvent()` (`SleeEndpointImpl.java:186`) takes the event as-is. USSD CONTINUE arriving twice → SBB executes `onEvent` twice → state machine may corrupt (e.g. timer armed twice). | 🟠 major |
| 5 | **Session stuck on upstream service die.** | `GrpcMenuResourceAdaptor.doCall()` at `GrpcMenuResourceAdaptor.java:111-131` submits the upstream call on `workerPool` and fires a `GrpcMenuResponseEvent`. If the upstream gRPC call hangs forever, no response event is ever fired and the SBB's session timer is the only thing that eventually frees the session. The example wiring relies on a 30 s timeout (`UssdSbbWiring.java:45`), but a 10 K-concurrent-stuck-session storm can blow the slot pool because the SBB entities are never released. | 🟠 major |
| 6 | **CMP snapshot taken AFTER the slot is recycled; lazy fields may be stale.** | `MicroSleeContainer.buildLocalObject()` removal listener (`MicroSleeContainer.java:1322-1359`) calls `cmpPersist()` on a `CmpBackedSbb` then releases the entity. The cluster path (`DistributedSbbEntityPool.release → persistSnapshot` at `DistributedSbbEntityPool.java:329-341`) takes a snapshot via the `@CmpField` getters at the moment of release. But `CmpBackedSbb` writes are lazy (`CmpAccessorInvoker` round-trips through the store on every `getXxx`/`setXxx`), so the in-memory SBB state held only in `EntityState.cmpFields` (`SbbEntityState.java:46`) is captured only when those fields were read via the accessors. Fields that were set but never re-read in this session may show stale defaults on reconstruction. | 🟠 major |
| 7 | **Cluster ACNF cache holds non-serializable ACI objects.** | `ClusteredActivityContextNamingFacility` cache mode is `DIST_SYNC` (`ClusteredActivityContextNamingFacility.java:92-94`). The class header at lines 44-55 explicitly acknowledges `ActivityContextInterface` is NOT `Serializable`. A cross-node lookup will throw `NotSerializableException` at runtime (acknowledged P5 follow-up). | 🟡 minor (documented) |
| 8 | **Timer-bridge ACI auto-bind can leak ghost entities.** | `SleeTimerSchedulerBridge.timerFireCallback` at `SleeTimerSchedulerBridge.java:103-113` routes a `TimerFiredEvent` onto `target.activityContext`. If the SBB has been removed but `target` still sits in the `targetsByTimerId` map (because `cancel` was never called), the event is fired onto an orphan ACI and the EventRouter has no attached SBB to dispatch to. | 🟡 minor |
| 9 | **`fireEvent()` can be invoked while RA state machine is STOPPING/INACTIVE but is rejected.** | `SleeEndpointImpl.fireEvent()` calls `stateMachine.assertCanFireEvent()` at `SleeEndpointImpl.java:196` which throws `IllegalStateException`. The RA's caller (HTTP/gRPC handler thread) has no outbox to fall back to. | 🟡 minor |
| 10 | **No convergence-key expiry.** | `InitialEventSelectorDispatcher.convergenceIndex` is a plain `ConcurrentHashMap` (`InitialEventSelectorDispatcher.java:76`). If the IES removal callback never fires (e.g. SBB killed mid-event before removal listener runs), the entry leaks until the JVM restarts. | 🟡 minor |

### 2.2 SBB death + incoming message flow (gap case)

When the slot is recycled (e.g. `EntitySlotPool.borrow()` runs out of
slots and `created.get() >= max` ⇒ an old slot is reused at
`EntitySlotPool.java:46-54`), the old entity's SBB POJO reference is
gone. `MicroSleeContainer.sbbs` no longer has the id.

```
RA → fireEvent → EventRouter.dispatch → dispatchWithTransaction
   → activityContext.getAttachedSbbs()                          [EventRouter.java:383]
       // ACI still has the stale attach from the removed entity
       → for sbbLocalObject in attached
            → localObject.isRemoved()? continue                  [line 392-394]
            → acceptsEvent? continue                              [line 401]
            → handler.onEvent(...)
```

If the removal listener ran (`MicroSleeContainer.java:1322-1359`),
`localObject.isRemoved()` returns `true` (set at
`SimpleSbbLocalObject.java:173`) and the event is skipped — silently.
**No log, no metric, no callback to the RA.** The user-visible outcome
is "the USSD session stopped responding" with no diagnostic.

### 2.3 IES-driven session create vs continue


### 2.7 Cluster manager interlock on entity removal

The IES auto-cleanup callback registered at
`InitialEventSelectorDispatcher.java:156-159` is fired by the
`SbbEntityPool.onEntityRemoved(...)` SPI declared at lines 65-71.
micro-jainslee's `VirtualThreadSbbEntityPool` does **not** implement
this SPI — it only exposes `acquire`/`release`/`findEntity`. The
callback therefore never fires for the in-memory pool. Only the
`FakePool` in the unit test
(`InitialEventSelectorDispatcherTest.java:46-74`) drives it manually.

Cross-references:
* `InitialEventSelectorDispatcher.removeConvergencesFor(entityId)` at
  `InitialEventSelectorDispatcher.java:169-174` is the explicit
  cleanup path, but no production code calls it.
* `CascadeRemover.entityLookup().releaseEntity(rootEntityId)` at
  `CascadeRemover.java:158` is the only production trigger for
  convergence cleanup — via the cascade path, not the simple-release
  path.

## 3. Gap analysis — 10 gaps with severity, impact & evidence

The gaps below are listed in priority order (severity × blast-radius).
Each one has:
* **Where** — concrete file:line
* **Impact** — what breaks for the USSD demo
* **Reproducer** — minimal sketch (no code written; conceptual only)

### GAP-SR-1 · SBB entity death between events is invisible to the RA 🔴 P0

* **Where:** `EventRouter.deliverEvent()` at `EventRouter.java:479-548`;
  the `findEntity(pool, ...)` call at line 480 falls back to inline
  delivery at line 483-489 when the pool has no entry.
* **Impact:** A USSD CONTINUE arriving after the HTTP SBB was removed
  (e.g. session timeout fired and `HttpServerSbb.onTimer` already
  called `wiring.failSession(...)`) will silently disappear.
* **Reproducer (sketch):**
  1. Open USSD session — HTTP SBB allocated, slot bound.
  2. Wait 30 s — `HttpServerSbb.onTimer` runs, calls
     `wiring.failSession → clearSession → local.remove()` →
     `RemovalListener.onRemoved` → `sbbEntityPool.release(entity)` →
     slot unbound (`EntitySlot.unbind` at `EntitySlot.java:94-100`).
  3. A late USSD CONTINUE arrives from the gRPC RA — fires onto the
     same ACI. `dispatchWithTransaction` iterates attached SBBs, every
     one returns `isRemoved()=true`, the for-loop body never runs.
* **Severity:** 🔴 critical — user-visible "session stops responding"
  with no log, no metric.

### GAP-SR-2 · No RA notification on entity removal 🔴 P0

* **Where:** `VirtualThreadSbbEntityPool.release*` at
  `VirtualThreadSbbEntityPool.java:200-217` and the
  `RemovalListener` lambda in `MicroSleeContainer.buildLocalObject`
  at `MicroSleeContainer.java:1322-1359`.
* **Impact:** HTTP RA keeps `UssdSessionStore` entry alive (separate
  process-local store), so polling clients see "PROCESSING" forever.
  gRPC RA has no way to cancel a pending upstream call bound to that
  session id.
* **Reproducer (sketch):**
  1. Open session — `sessionStore.open(sessionId)` at
     `UssdSbbWiring.java:115` (record in `PROCESSING` state).
  2. gRPC upstream call hangs (network partition).
  3. Session timer fires — `wiring.failSession` →
     `sessionStore.fail(sessionId, msg)` — store correctly updates.
     But if removal was driven by an *external* cause (operator
     command, OOM eviction, hot redeploy), the store is never
     informed.
* **Severity:** 🔴 critical — leaks in `UssdSessionStore` and RA
  worker thread pools indefinitely.

### GAP-SR-3 · Out-of-order messages are dropped, not queued 🟠 P1

* **Where:** `InitialEventSelectorDispatcher.resolveTarget` at
  `InitialEventSelectorDispatcher.java:139-144`. The spec §7.5.5 rule
  says "non-initial event with no matching entity → silently dropped".
  micro-jainslee implements this literally with no buffer.
* **Impact:** A CONTINUE message that arrives BEFORE the BEGIN due to
  network reordering is lost. User retries, the second CONTINUE
  succeeds, but the SBB has no way to know a message was dropped.
* **Reproducer (sketch):**
  1. CONTINUE arrives first — IES returns `null` (line 142).
  2. EventRouter.routeEvent receives the event but the legacy
     `attachRootSbbViaInitialEventSelector` branch at
     `MicroSleeContainer.java:1080-1082` returns early because
     `!selector.isInitialEvent()`.
  3. `dispatchWithTransaction` runs with `attachedSbbs.isEmpty()` —
     no-op.
* **Severity:** 🟠 major — requires app-level retry; silently broken
  end-to-end.

### GAP-SR-4 · Duplicate / retried messages processed twice 🟠 P1

* **Where:** `SleeEndpointImpl.fireEvent()` at
  `SleeEndpointImpl.java:186` accepts the event object as-is; no
  dedup window.
* **Impact:** USSD gateways commonly retry CONTINUE messages on no
  response. Without dedup, the SBB processes both, fires duplicate
  `GrpcMenuRequestEvent` (with `GrpcMenuResourceAdaptor.requestMenu`
  at line 82-109 firing two upstream gRPC calls), and may end with
  two `UssdCompleteEvent` callbacks.
* **Severity:** 🟠 major — duplicates user-visible menu items, double
  bills if any chargeable operation, race in CMP writes.

### GAP-SR-5 · No timeout / cancellation on upstream service die 🟠 P1

* **Where:** `GrpcMenuResourceAdaptor.doCall` at
  `GrpcMenuResourceAdaptor.java:111-131`; only path off the worker
  thread is the session-timer in `HttpServerSbb`.
* **Impact:** A dead gRPC upstream leaves the slot occupied until the
  session timer fires. At 30 s timeout and 4 K concurrent stuck
  sessions, the slot pool runs out → new sessions can't be opened.
* **Reproducer (sketch):**
  1. Open 4096 sessions, each fires `requestMenu`.
  2. Simulate gRPC upstream hang by replacing the client with a
     `CompletableFuture` that never completes.
  3. After 30 s all 4096 session timers fire and `failSession` runs,
     but until then no slot is available for new sessions.
* **Severity:** 🟠 major — capacity collapse; non-obvious because the
  log only shows session timeouts, not pool exhaustion.

### GAP-SR-6 · Cluster snapshot loses attached-ACI set 🟠 P1

* **Where:** `DistributedSbbEntityPool.takeSnapshot` at
  `DistributedSbbEntityPool.java:266-268` passes an empty
  `LinkedHashSet` for `attachedAciNames`; `SbbEntitySnapshot` at
  `SbbEntitySnapshot.java:71` carries the field but it is never
  populated.
* **Impact:** Cross-node failover reconstructs the SBB with its CMP
  fields but does NOT re-attach it to any ACI. The next event on the
  ACI finds no attached SBB and the legacy
  `attachRootSbbViaInitialEventSelector` path creates a fresh entity,
  effectively losing the session.
* **Severity:** 🟠 major — breaks cluster failover for any stateful
  SBB.

### GAP-SR-7 · IES auto-cleanup callback never fires for the in-memory pool 🟠 P1

* **Where:** `InitialEventSelectorDispatcher.pool.onEntityRemoved(...)`
  at `InitialEventSelectorDispatcher.java:156-159` is the registration
  point, but `VirtualThreadSbbEntityPool` does not implement the
  `SbbEntityPool` SPI declared at lines 59-71.
* **Impact:** When a non-cascade removal occurs (`SimpleSbbLocalObject.remove`
  at `SimpleSbbLocalObject.java:163-190` calls
  `RemovalListener.onRemoved` which only fires cascade removal), the
  convergence entry in `InitialEventSelectorDispatcher.convergenceIndex`
  (`InitialEventSelectorDispatcher.java:76`) leaks.
* **Severity:** 🟠 major — unbounded memory growth over a long-lived
  JVM with many short sessions.

### GAP-SR-8 · Cluster ACNF cache stores non-Serializable ACIs 🟡 P2

* **Where:** `ClusteredActivityContextNamingFacility.bind` at
  `ClusteredActivityContextNamingFacility.java:127-132` calls
  `cache.put(name, aci)` where `aci` is an `InMemoryActivityContext`
  that does NOT implement `Serializable`. The header at lines 44-55
  explicitly acknowledges this.
* **Impact:** A cross-node `lookup(name)` will throw
  `NotSerializableException` at runtime. Affects cluster mode only.
* **Severity:** 🟡 minor (documented as P5 follow-up) — R&D code
  never crosses nodes, but the wiring is misleading.

### GAP-SR-9 · Timer-bridge leaks ghost entities 🟡 P2

* **Where:** `SleeTimerSchedulerBridge.timerFireCallback` at
  `SleeTimerSchedulerBridge.java:103-113`. `targetsByTimerId.remove`
  at line 106 returns null when the timer was cancelled after the
  record fired (rare race) — no harm. But if `cancel` is never
  called and the SBB is removed, the bridge holds a stale reference
  to the activity context.
* **Impact:** A `TimerFiredEvent` is routed to an ACI whose attached
  SBBs list is empty (or contains only removed SBBs) — silent no-op
  in `dispatchWithTransaction`.
* **Severity:** 🟡 minor — small leak; difficult to detect in
  production.

### GAP-SR-10 · No convergence-key expiry 🟡 P2

* **Where:** `InitialEventSelectorDispatcher.convergenceIndex` at
  `InitialEventSelectorDispatcher.java:76` is a plain
  `ConcurrentHashMap`; no time-based eviction.
* **Impact:** Complements GAP-SR-7. Even with proper auto-cleanup,
  edge cases (SBB killed mid-event, hot redeploy) leave entries.
* **Severity:** 🟡 minor — only matters at very high session churn.

### GAP-SR-11 · example-quarkus UssdSbbWiring.java has broken imports 🟡 P2

* **Where:** `UssdSbbWiring.java` lines 22-23 import
  `com.microjainslee.ra.grpc.GrpcMenuResourceAdaptor` and
  `com.microjainslee.ra.http.HttpIngressResourceAdaptor`, but the
  actual classes live in `com.example.ussddemo.quarkus.ra`.
* **Impact:** The example will not compile as-shipped. Reference for
  how to wire the wiring layer is lost; every copy of the example is
  broken.
* **Severity:** 🟡 minor — R&D only, but defeats the point of the
  example.

## 6. Acceptance criteria

### 6.1 Per-sprint acceptance

**S6 (observability & removal notification):**
* `MicroSleeContainer.getEntityRemovalCount()` returns N after N entities have been removed.
* `EventRouter.getMissingEntityCount()` returns M after M events were routed to removed entities.
* `UssdSessionStore.get(sid).getStatus()` is `FAILED` within 100 ms of an entity removal driven by any source (timer, exception, SBB self-remove, hot redeploy).
* All existing unit tests still pass (no regression).

**S7 (re-creation & replay):**
* A test that kills an entity then fires a CONTINUE 5 ms later observes the SBB onEvent invocation with the last CMP snapshot restored.
* The rehydrate cache never exceeds `MicroSleeConfiguration.rehydrateCacheSize` (default 64 K) entries.
* The recovery path does NOT loop infinitely on a stream of events for the same missing entity (bounded retry count per convergence key).

**S8 (ordering + dedup):**
* A CONTINUE arriving BEFORE BEGIN is buffered and delivered within 100 ms of the BEGIN.
* A duplicate CONTINUE (same `(convergence, seq)`) within the dedup window is dropped with a debug log and the dedup-hit metric increments.
* After dedup window expiry, a re-transmitted CONTINUE is delivered normally (no permanent suppression).

**S9 (cluster + example):**
* Cluster failover test: kill node A holding session `s1`; on node B, fire CONTINUE; verify SBB on B receives the event with all CMP fields intact AND is attached to the original ACI name.
* `example-quarkus` compiles (`mvn -pl example/example-quarkus compile`) and the existing smoke tests pass.
* IES auto-cleanup test: open 1000 sessions, close them all, verify `InitialEventSelectorDispatcher.activeConvergenceCount() == 0` after the close phase.

### 6.2 Cross-cutting acceptance (end-of-S9)

* All 246 existing unit tests + 57 new tests pass in CI on every push.
* Memory footprint of a 10 K-session long-running JVM does not grow by more than 5 % per hour.
* A chaos test that randomly kills 50 % of entities per minute observes:
  * `EventRouter.getMissingEntityCount()` increments, but
  * `EventRouter.getRehydratedCount()` >= 80 % of `MissingEntityCount` (recovery rate).
  * Zero `OutOfOrderBuffer` overflow exceptions in steady state.

### 6.3 Documentation acceptance

* `docs/WIRING_GUIDE.md` updated with the new `notifyEntityRemoved` callback contract.
* `docs/MICRO-JAINSLEE-GUIDE.md` adds a "Session recovery" section explaining the rehydrate flow.
* `docs/micro-jainslee-cmp-production-roadmap.md` cross-references this report under §6 (Perfect Core mapping) so future maintainers find both docs together.
